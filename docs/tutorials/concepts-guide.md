# دليل المفاهيم — صوري

> شرح موجز للمفاهيم التقنية المستخدمة في المشروع، مناسب لمبتدئ يقرأ الدروس لأول مرة.

---

## 1. Next.js

**ما هو:** إطار عمل مبني على React يجمع بين واجهة الصفحة الواحدة (SPA) وتقديم الصفحات من الخادم (SSR) ومسارات API داخل المشروع نفسه.

**لماذا في صوري:** نستخدم **App Router** (مجلد `app/`) لتعريف الصفحات ومسارات API. كل مجلد `page.tsx` يصبح صفحة، وكل `route.ts` داخل `api/` يصبح نقطة نهاية HTTP. لا حاجة لخادم Express منفصل — Next.js يوفّر الخادم وبناء الصفحات.

**تشبيه:** Next.js مثل مطعم يقدّم الطلب (الصفحة) من المطبخ (الخادم) ويقدّم أيضاً طلبات خاصة (API) من نفس المطبخ.

---

## 2. TypeScript

**ما هو:** لغة برمجة مبنية على JavaScript تضيف **أنواعاً ثابتة** (strings، numbers، objects ذات شكل معروف). المحرر والمترجم يكتشفان الأخطاء قبل التشغيل.

**لماذا في صوري:** كل ملفات المشروع `.ts` أو `.tsx`. واجهات مثل `User` و `Photo` معرّفة في `types.ts` وتُستخدم في API والمكونات. يقلّل الأخطاء عند تغيير شكل البيانات ويُسهّل القراءة.

**تشبيه:** TypeScript كقائمة محتويات ثابتة لكل صندوق: إذا وضعت شيئاً غير مذكور في القائمة، يُنبّهك النظام.

---

## 3. MongoDB و Mongoose

**MongoDB:** قاعدة بيانات NoSQL تخزّن المستندات (JSON-like) في مجموعات (collections). لا تحتاج جداول ثابتة مسبقاً.

**Mongoose:** مكتبة لـ Node.js تربط JavaScript بـ MongoDB عبر **نماذج** (Schemas). تحدد الحقول، الأنواع، والفهارس، وتوفّر دوال مثل `find` و `save`.

**لماذا في صوري:** ثلاث مجموعات: `users`، `photos`، `likes`. Mongoose يتحقق من شكل البيانات ويُبقي العلاقات (مراجع ObjectId) واضحة.

**تشبيه:** MongoDB خزانة ملفات مرنة؛ Mongoose هو نظام تصنيف وملصقات يحدد أين يُوضَع كل نوع من الملفات.

---

## 4. نمط المستودعات (Repository Pattern)

**ما هو:** طبقة وسيطة بين منطق التطبيق (مثل API Routes) وقاعدة البيانات. بدلاً من استدعاء Mongoose من كل مسار، نستدعي دوال مثل `userRepo.findById()` أو `photoRepo.findPublicFeed()`. تفاصيل الاستعلامات مخفية داخل المستودع.

**لماذا في صوري:** فصل واضح: مسارات API لا تعرف تفاصيل Mongoose. يسهّل استبدال مصدر البيانات وكتابة اختبارات وهمية للمستودعات.

**تشبيه:** المستودع وكيل بينك وبين المستودع الفعلي: تطلب "صورة رقم 5" وهو يعرف كيف يذهب ويجلبها من الرف المناسب.

---

## 5. JWT (JSON Web Token) والجلسة في صوري

**ما هو:** توكن مشفّر يحمل بيانات (مثل معرّف المستخدم). الخادم يوقّعه بمفتاح سري (`JWT_SECRET`). الخادم لا يخزّن جلسات في الذاكرة — التحقق من التوقيع يكفي لمعرفة هوية الطلب.

**لماذا في صوري (التطبيق الحالي):** بعد تسجيل الدخول أو التسجيل، يُخزَّن الـ JWT في **Cookie HttpOnly** باسم `auth-token` (يضبطه الخادم في `Set-Cookie`). JavaScript **لا يقرأ** التوكن — يقلل خطر سرقته عبر XSS. المتصفح يُرسِل الـ cookie تلقائيًا مع طلبات نفس النطاق. مسارات API تستدعي `authenticateRequest(request)` التي تقرأ التوكن من الـ **cookie أولاً**، ثم من `Authorization: Bearer` كبديل (اختبارات أو عملاء برمجيون). **Next.js Proxy** (`src/proxy.ts`) يتحقق من وجود الـ cookie قبل عرض صفحات مثل `/my-photos` و`/profile`. تسجيل الخروج: `POST /api/auth/logout` يمسح الـ cookie.

**تشبيه:** JWT تذكرة موقّعة؛ في التطبيق تُحفظ التذكرة في خزنة لا يصل إليها JavaScript (HttpOnly)، بينما البوابة (المتصفح) يمرّرها تلقائيًا للخادم عند كل زيارة.

---

## 6. استراتيجية التخزين (Strategy Pattern)

**ما هو:** نمط تصميم يغيّر سلوكاً (مثل "أين أرفع الملف") دون تغيير الكود الذي يستدعيه. واجهة واحدة (`uploadFile`، `deleteFile`) وتنفيذات متعددة (محلي، Cloudinary، S3). يُختار التنفيذ عند التشغيل حسب متغير البيئة.

**لماذا في صوري:** `STORAGE_TYPE=local` للتطوير و `cloudinary` أو `s3` للإنتاج. مسارات الرفع والحذف تستدعي `getStorageService()` فقط؛ لا تعرف أين الملف فعلياً.

**تشبيه:** نفس "طلب توصيل" واحد، لكن شركة التوصيل تتغير حسب العنوان (محلي، سحابة أ، سحابة ب).

---

## 7. getUserMedia و الكاميرا

**ما هو:** واجهة متصفح (Web API) تطلب إذن الوصول إلى الكاميرا أو الميكروفون. تُرجع `MediaStream` يمكن ربطه بعنصر `<video>` ثم التقاط إطار (مثلاً عبر `<canvas>`) وتحويله إلى ملف صورة.

**لماذا في صوري:** خطاف `useCamera` يستدعي `navigator.mediaDevices.getUserMedia({ video: true })`. مكوّن `CameraCapture` يعرض البث ويحوّل اللقطة إلى `File` لرفعها كصورة منشور أو صورة شخصية. في أجهزة لا تدعم getUserMedia (مثل بعض إصدارات iOS) يُستخدم fallback (مثل input file مع capture).

**تشبيه:** getUserMedia طلب إذن لفتح كاميرا الجهاز؛ البرنامج يقرأ الصورة الحية ثم يلتقط لقطة ويحوّلها إلى ملف.

---

## 8. Vitest و Testing Library

**Vitest:** إطار اختبار سريع متوافق مع واجهة Jest، مصمم لمشاريع Vite/TypeScript. يشغّل ملفات `*.test.ts` أو `*.test.tsx`.

**Testing Library:** مجموعة أدوات لاختبار مكونات React من منظور المستخدم (العثور على عناصر بالنص أو الدور، محاكاة الضغط والإدخال). التشجيع على عدم الاعتماد على تفاصيل التنفيذ الداخلية.

**لماذا في صوري:** `vitest.config.ts` يحدد البيئة `jsdom` وملف `setup.ts`. الاختبارات تتحقق من دوال المصادقة، المستودعات، المكونات، والصفحات. رسائل `describe` و `it` بالإنجليزية؛ النص المعروض في الواجهة يبقى عربياً في الاختبارات.

**تشبيه:** Vitest المنصة التي تشغّل الاختبارات؛ Testing Library الطريقة التي تسأل بها "هل يرى المستخدم هذا النص؟ وهل الزر يعمل عند الضغط؟".

## ملخص سريع

| المفهوم                  | الدور في صوري                                           |
| ------------------------ | ------------------------------------------------------- |
| Next.js                  | إطار الصفحات و API الموحّد                              |
| TypeScript               | أنواع ثابتة لكل البيانات والواجهات                      |
| MongoDB + Mongoose       | قاعدة البيانات والنماذج (User، Photo، Like)             |
| Repository Pattern       | طبقة وصول بيانات موحدة للمستودعات                       |
| JWT + Cookie HttpOnly    | مصادقة عديمة الحالة؛ التوكن في cookie آمن + تحقق في API |
| Strategy Pattern         | تبديل التخزين (local/cloudinary/s3) دون تغيير API       |
| getUserMedia             | التقاط الصور من كاميرا الجهاز                           |
| Vitest + Testing Library | تشغيل الاختبارات واختبار المكونات                       |
